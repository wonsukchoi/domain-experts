# Playbook

## Case-currency check worksheet

Fill before teaching any previously-used case, not just when a case "feels old."

| Field | Value |
|---|---|
| Case | SaaS Pricing Model Shift ("MeridianOps: Per-Seat or Usage-Based?") |
| Case originally written | 4 years ago |
| Industry/market covered | B2B SaaS subscription pricing, mid-market segment |
| Regulatory/market conditions at time of writing | Per-seat pricing dominant; usage-based pricing viewed as a niche approach used mainly by infrastructure vendors |
| What's changed since | 3 of the case's 5 cited "per-seat leader" competitors have since shifted partly or fully to usage-based or hybrid pricing; one of the 3 named comparison companies was acquired 18 months ago |
| Does the changed context affect the core decision being taught? | Yes — the case frames usage-based pricing as the riskier, less-proven option; that framing is no longer accurate given the shift among named competitors |
| Action | Add a 1-page current-conditions addendum listing the 3 competitor pricing shifts and the acquisition, distributed before the case discussion, not folded silently into the original case text |
| Addendum drafted and dated | Y — dated to current term, cites source for each competitor shift |
| Re-check due | Next offering of the course, or sooner if a named company in the case has further pricing or ownership changes |

**Rule:** run this check every time a case is reused, not only on a fixed multi-year cycle — a named competitor in the case can change its relevant facts (pricing model, ownership, market position) well before a scheduled review date, and the addendum needs a citable source per changed fact, not a general "things have changed" note.

## Reasoning-based grading rubric example

Same pricing case as above. Assignment: recommend per-seat, usage-based, or hybrid pricing for MeridianOps, with written justification.

| Criterion | Weight | What's assessed |
|---|---|---|
| Problem framing | 15% | Does the submission correctly identify the actual trade-offs at stake (revenue predictability vs. customer cost-alignment vs. sales-motion complexity), not just restate the prompt? |
| Framework application | 25% | Is a taught pricing-strategy framework (e.g., value-metric alignment) applied to this specific case's numbers, not recited in the abstract? |
| Evidence use | 25% | Does the submission use the case's actual data (the 3 competitor shifts, the acquisition, MeridianOps' own usage-variance figures) rather than generic pricing-strategy claims that could apply to any company? |
| Trade-off treatment | 25% | Does the submission acknowledge what its recommendation gives up, not just what it gains? (E.g., a usage-based recommendation should address the sales-forecasting complexity it introduces.) |
| Recommendation coherence | 10% | Does the final recommendation follow from the analysis presented, rather than arriving at a conclusion the analysis doesn't support? |

**Conclusion is not a scored criterion.** The rubric has no line item for "chose the pricing model the instructor prefers."

Sample outcomes from one section (24 submissions):

| Recommendation chosen | Submissions | Top-two-tier scores (of that group) |
|---|---|---|
| Usage-based | 9 | 7 |
| Hybrid | 10 | 8 |
| Per-seat (retain current) | 5 | 3 |

Reconciling: 18 of 24 (75%) scored in the top two tiers, spread across all 3 recommendations — including 3 of 5 per-seat submissions, which is the option that requires the strongest trade-off argument given the competitor shifts documented in the addendum. A single-answer key graded to "hybrid" (the instructor's own view going in) would have scored only the 10 hybrid submissions well regardless of reasoning quality, misgrading up to 8 of the other 14 defensible submissions.

## Framework-vs-opinion flagging convention

Use a consistent verbal or written marker so students can tell which mode is in play — don't rely on tone alone.

| Signal phrase | When to use |
|---|---|
| "This is a taught framework: ..." | Introducing a named, citable analytical model (e.g., value-metric pricing alignment, Porter's five forces) |
| "This next part is my own experience, not something you'll find in the reading: ..." | Sharing a personal anecdote, a past employer's practice, or a view not backed by an established framework |
| "Multiple frameworks disagree on this point; here's how each would treat it: ..." | When two taught frameworks would reach different conclusions on the same case fact |
| No flag used, presented as settled | Reserved for facts (case data, dates, figures), never for a judgment call |

**Rule:** if a student can't tell, from the framing alone, whether a claim is a citable model or an instructor's personal take, the claim needs a flag — this applies most often right after a framework is taught, when the instructor's own application example can blur into "the" correct application rather than "an" application.

## Ambiguity-preservation checklist for case design

Before finalizing a case problem's dataset, check that it still reflects real incomplete-information conditions:

- [ ] At least one key input is a range or estimate ("6-18 months," "limited to 2 prior data points"), not a single precise figure, where real decisions of this type would face that same uncertainty.
- [ ] At least one plausible response from a third party (competitor, regulator, partner) is stated as uncertain rather than given.
- [ ] More than one course of action can be defended using only the data provided — verify this by drafting a short defensible argument for at least 2 different conclusions before assigning the case.
- [ ] The case does not contain a single data point that, alone, makes one option obviously dominant (that's a sign the case has been oversimplified toward a single answer).
- [ ] For an introductory case meant to teach one framework cleanly, ambiguity is deliberately narrowed rather than maximized — confirm the case's teaching goal (clean framework introduction vs. realistic messiness) before adding uncertainty for its own sake.

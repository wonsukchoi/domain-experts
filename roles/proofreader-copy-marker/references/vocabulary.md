# Proofreader / Copy Marker — Vocabulary

### Stet
A proofreader's mark meaning "let it stand" — cancels a previous correction mark and restores the original text. Indicated by dots under the struck-through text plus "stet" written in the margin.
**In use:** "The line-editor changed 'grey' to 'gray,' but this is UK-market copy — stet the original spelling."
**Common misuse:** People assume a struck-through word is automatically deleted; a stet mark specifically reverses that, and missing it means re-deleting text that was deliberately restored.

### Query
A flagged question routed back to the author or editor rather than a direct correction — used when something looks wrong but the proofreader can't independently verify it.
**In use:** "There's a query on page 19 about the SKU number — it needs the product team to confirm before we can close out this page."
**Common misuse:** Treating a query as equivalent to a correction ("I flagged it, so it's handled") — a query is only resolved once someone with the authority to confirm the fact actually answers it.

### Blueline (press-ready proof)
The final proof stage representing exactly what will print, historically a blue-toned photographic proof, now usually a press-ready PDF. The last point where a change is inexpensive; changes after this stage typically require replating.
**In use:** "We're at blueline — anything caught now is free, anything caught after this is a reprint."
**Common misuse:** Treating a blueline review with the same casualness as an early galley read — it's the last free checkpoint, not just another round.

### Widow / orphan
A widow is a single short line of a paragraph left alone at the top of a new page or column; an orphan is a single word or short line left alone at the bottom of a page or column, separated from its paragraph. Both are typographic defects introduced or fixed by small text-length changes.
**In use:** "The price correction pushed the caption to create an orphan — one word by itself on line one of the next column."
**Common misuse:** Assuming a small text edit (a one-word correction) can't affect layout — even a short fix can reflow a paragraph enough to create or resolve a widow/orphan several lines away.

### Style sheet (proofreading context)
A running, project-specific record of every spelling, capitalization, hyphenation, and formatting decision made for a document, built cumulatively as questions arise — distinct from a published house style guide, which sets defaults the style sheet extends.
**In use:** "Check the style sheet before flagging 'e-mail' as wrong — we decided on the hyphenated form for this project in round one."
**Common misuse:** Relying on memory of "how we did it last time" instead of the actual style sheet, which is the only authoritative record when a project has made a nonstandard choice.

### Regression (proofing context)
A new error introduced by a correction, distinct from an error that survived uncaught from an earlier round. Identifying something as a regression matters because it points at the correction process itself as the source, not the original content.
**In use:** "That's a regression — the hyphenation fix on page 22 is what introduced the double space, it wasn't there in round one."
**Common misuse:** Logging a regression as if it were an original missed error, which hides a pattern of the correction process itself introducing defects.

### Mechanical error vs. factual error
A mechanical error is objectively verifiable from the page alone (spelling, punctuation, spacing, formatting) and gets corrected directly. A factual error (a price, date, name, statistic) requires an external source to confirm and gets queried, not corrected on sight.
**In use:** "The typo is mechanical, fix it. The price discrepancy is factual — query it, don't guess which number is right."
**Common misuse:** Applying mechanical-error confidence to factual claims — "correcting" a number that looked wrong but was actually right, because it wasn't independently checked against a source.

### Systemic error / root-cause flag
A pattern of the same defect recurring three or more times across a document, indicating a template, setting, or process cause rather than independent unrelated mistakes.
**In use:** "Four instances of the same bad hyphenation break — that's systemic, flag the setting instead of hand-fixing each one."
**Common misuse:** Treating each recurrence as a fresh, isolated find, which burns time on symptom-fixing and leaves the actual cause (and future recurrences) unaddressed.

### Copyfitting
Adjusting text length (via cuts, tracking, or type size) to fit a fixed layout space without a full re-edit — a mechanical/typographic operation distinct from an editorial trim.
**In use:** "The caption is two characters too long for the box — that's a copyfitting problem, not a content edit."
**Common misuse:** Confusing copyfitting with editing; a proofreader can flag a copyfitting problem but shouldn't be rewriting the sentence to solve it — that's the editor's or writer's call.

### Dead copy / live copy
Dead copy is the previously approved, corrected version used as the reference baseline; live copy is the current proof being checked against it. Distinguishing the two matters when verifying that only the intended changes were made.
**In use:** "Compare live copy against dead copy line by line on page 19 — we need to confirm nothing else shifted besides the price fix."
**Common misuse:** Proofreading live copy in isolation without a dead-copy comparison, which is how unintended changes (a regression) go undetected.

### Proof round / pass number
A sequential count of proofing cycles on a document, tracked explicitly because the required scrutiny and expected error type changes with each round (first-read accuracy vs. correction-verification vs. final zero-tolerance check).
**In use:** "This is round three — we shouldn't be finding new mechanical errors at this stage, only confirming the round-two fixes stuck."
**Common misuse:** Treating every round with identical scope and effort instead of adjusting focus — round two exists specifically to catch regressions, not to re-read for brand-new errors from scratch.

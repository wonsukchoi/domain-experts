## Быстрый старт

```sh
npx domain-experts match "review this vendor contract like a lawyer"
npx domain-experts add lawyer-contracts   # installs into ./.claude/skills/
```

Установка не нужна — `npx` подтягивает пакет прямо из npm. Пользуетесь часто? Поставьте `npm install -g domain-experts` и убирайте `npx`.

**Используете Claude Code, Codex, Gemini CLI, Cursor, Windsurf, Roo Code или Amp?** Команда `npx domain-experts command --tool <id>` устанавливает slash-команду `/domain-expert` — перезапустите сессию и выполните `/domain-expert review this vendor contract`. Она сама подберёт нужную роль, загрузит её и станет рассуждать как эта роль за один шаг, без ручного танца `match`/`add`.

Или вообще пропустите ручной шаг: загрузите [`skills/domain-expert-router/SKILL.md`](./skills/domain-expert-router/SKILL.md) один раз, и ваш агент сам определит, какой эксперт нужен для задачи, автоматически подтянет полный контекст роли и честно скажет, если подходящей роли ещё нет, вместо того чтобы импровизировать. Вы продолжаете работать, а нужная экспертиза появляется сама.

## «А разве нельзя просто попросить Claude вести себя как CFO?»

Можно — и вы получите поверхностную имитацию: усреднение всех должностных инструкций интернета, заново сгенерированное с нуля в каждой сессии, каждый раз разное и никем не проверенное.

```
 ── prompt: "act as a CFO" ───────────┬── role: financial-manager ───────────
                                      │
  "I'd start by monitoring cash       │  "DSO went 48 → 56 days with no
   flow and key financial metrics,    │   billing-terms change. Show me the
   ensuring alignment between…"       │   five largest invoices past 60 days
                                      │   — and reconcile bookings to the
                                      │   change in deferred revenue, because
                                      │   flat deferred + 'record bookings'
                                      │   don't coexist."
 ─────────────────────────────────────┴───────────────────────────────────────
```

Разница конкретно в следующем:

- **Непроизводимое содержание.** Каждая роль обязана пройти тест на непроизводимость: в ней не должно быть ничего, что можно заново сгенерировать из одного только названия должности. Остаётся то, что промпт не способен выдать на лету по запросу — числовые пороги тревожных сигналов, рыночные диапазоны для переговоров, разобранные примеры со сходящейся арифметикой, запасные позиции в порядке приоритета.
- **Контроль качества, а не единичная генерация.** Роли создаются через многоступенчатый конвейер ([`AUTHORING.md`](./AUTHORING.md)) — см. диаграмму ниже. Однострочный промпт ничего подобного не проходит.
- **Структура, закреплённая CI.** Каждый PR проходит через [`scripts/lint_roles.py`](./scripts/lint_roles.py): схему, обязательные разделы, работоспособность ссылок, запрещённые «водянистые» фразы, полноту тревожных сигналов, реальные числа в разобранном примере. Обобщённый текст в стиле должностной инструкции проваливает сборку.
- **Это накапливается.** Ваш промпт, написанный на ходу, исчезает вместе с сессией. Эти файлы накапливают правки практиков, проходят по лестнице зрелости (`draft` → `reviewed-by-practitioner` → `mature`) и версионируются по спецификации (`spec: 2` отмечает роли, соответствующие текущей планке), становясь лучше с каждым PR. Исправления доходят до всех.
- **Экономия токенов заложена в дизайн.** Каждая роль — это компактное ядро рассуждений (`SKILL.md`) плюс глубина по требованию (`references/`). Агент платит за глубину только тогда, когда задача действительно её требует:

```
roles/financial-manager/
├─ SKILL.md            ◀ always loaded · identity, first principles,
│                        heuristics, worked example with real numbers
└─ references/         ◀ loaded on demand
   ├─ artifacts.md       filled 13-week cash forecast, board slide, scenarios
   ├─ red-flags.md       DSO +15% QoQ · GM −200bps · headroom <20% …
   └─ vocabulary.md      bookings vs billings vs revenue vs ARR …
```

### Так в чём же настоящий защитный ров?

Честное возражение: ничто из вышесказанного не мешает кому-то сделать `git clone` этого же репозитория и выдать его за свой собственный продукт — лицензия MIT, никакой привязки к контенту. Честный ответ: набор файлов — не ров. Трудно скопировать именно машину, которая продолжает всё это производить и исправлять:

- **Конвейер, а не результат.** Скопировать 97 файлов можно одной командой. А вот цикл создания с состязательной критикой → 9-критериальным рубриком → блокировкой линтером ([`AUTHORING.md`](./AUTHORING.md)), который продолжает их производить и исправлять, скопировать нельзя — форк наследует сегодняшний снимок, а не завтрашние правки.
- **Стандарт, а не база данных.** `SKILL.md` уже работает в 30+ агентских инструментах. Быть крупнейшей библиотекой в переносимом открытом формате — это позиция дистрибуции, а не позиция контента: ценность в том, чтобы быть тем ответом по умолчанию, который люди находят, а не в самих байтах.
- **Проверено, а не заявлено.** Любой конкурент может сказать «написано экспертами». Немногие могут прямо у вас на глазах запустить `python3 evals/run_evals.py` и показать 13 побед из 15 в контрфактических сравнениях. Здесь доверие измеряется и воспроизводимо, а не декларируется.
- **Актуальность бьёт параметрическую память.** Даже если будущая модель обучится на публичном тексте этого репозитория, эти знания замрут на дате отсечки обучения. Исправления в этом репозитории выходят в тот же день, когда их подаёт практик — без цикла переобучения, с версионированием и указанием источника.
- **Дисциплина покрытия.** Костяк из 1016 профессий O*NET заставляет систематически закрывать длинный хвост (funeral-home-manager, wind-energy-operations-manager), до которого оппортунистический конкурент, курирующий только трендовые роли, просто не доберётся.
- **Бесплатное и переносимое побеждает подписку.** Это не конкурирует со счётом за вашу LLM — за инференс вы платите в любом случае. Это конкурирует с закрытым вертикальным SaaS («AI Legal Advisor» за $99/мес): такие продукты не могут тягаться с бесплатным, форкаемым и запускаемым на локальной модели решением без постоянных затрат.

Пока при 97 ролях и небольшой базе контрибьюторов это ещё не ров — это траектория. Ставка в том, что общий фонд будет накапливаться быстрее, чем любой форк сможет за ним угнаться, как только достаточное число практиков начнёт подавать правки вместо того, чтобы писать промпты с нуля в каждой сессии.

## Видение — один человек, все эксперты

```
                              ┌───────────────────────┐
                              │   Y O U  +  A G E N T  │
                              └───────────┬───────────┘
                                          │
             ┌──────────────┬────────────┼────────────┬──────────────┐
             │              │            │            │              │
             ▼              ▼            ▼            ▼              ▼
        lawyer-        financial-    data-        marketing-    clinical-
        contracts      manager       scientist    strategist    research-
        │              │             │            │             coordinator
        redline the    defend the    read the     kill the      │
        MSA            runway call   A/B test     dead channel  flag the
                                      right                      deviation

        no résumé screened. no calendar. no invoice. no waiting room.
        just: which role does this task need — load it — reason as it.
```

Сейчас, чтобы хорошо сделать что-то за пределами вашей собственной специализации, нужно нанимать, заключать контракт или искать контакты — находить юриста, ждать место в календаре CFO, платить по ставке маркетингового стратега. Это трение — налог на каждого соло-основателя, каждую небольшую команду, каждого человека, который сталкивается с проблемой вне своей экспертизы. Большинство людей просто не делают эту вещь — или делают её плохо.

Человек с подпиской на AI — или вообще с локальной моделью, без всякой подписки — и этим репозиторием этот налог не платит. Загрузите реальную логику рассуждений CFO для решения по runway. Загрузите playbook юриста по контрактам для правки договора. Загрузите суждение координатора клинических исследований для отклонения от протокола. Меняйте роли под каждую задачу, по требованию, платя лишь за инференс вместо найма. Один человек, один агент — накопленный опыт принятия решений сотен профессий.

```
   ─────────────────────────────────────────────────────────────
    1 person + 1 agent + N roles  >  the org chart it replaces
   ─────────────────────────────────────────────────────────────
```

Это и есть настоящая конечная цель, а не просто любопытный эксперимент: барьер между «мне нужен эксперт» и «он у меня есть» рушится. Это становится тем правдивее, чем больше покрытие — сегодня 92 роли против 1016 отслеживаемых профессий; дорожная карта существует для того, чтобы этот разрыв сокращался, а не оставался интересным вечно.

Это не заменяет суждение, ответственность или лицензирование там, где по закону они обязаны оставаться за человеком — каждая регулируемая роль (право, медицина, финансы) прямо об этом говорит. Это заменяет трение от отсутствия доступа к самим рассуждениям.

```
you ─── "review this vendor contract"
              │
              ▼
   ┌──────────────────────┐        ┌─ roles/lawyer-contracts/ ──────────────┐
   │  domain-expert       │        │                                        │
   │  router              │───────▶│  SKILL.md      the reasoning core      │
   │  (finds the expert   │        │  references/                           │
   │   your task needs)   │        │   ├─ clause-playbook.md  fallbacks     │
   └──────────────────────┘        │   ├─ red-flags.md        smell tests   │
                                   │   └─ vocabulary.md       terms of art  │
                                   └────────────────────┬───────────────────┘
                                                        │
                                                        ▼
                                     agent reasons like a senior
                                     contracts attorney — thresholds,
                                     market positions, redline language
```

## Как создаются роли

```
  named sources        draft to        adversarial         revise        score vs
  books · standards ─▶ AUTHORING ──▶  critique by a  ──▶  or contest ─▶  9-criterion ──▶ ship
  practitioners        spec           separate model      each defect    rubric
                                                                            │
                            below 14/18, or any zero:  ◀────────────────────┘
                            loop (max 2) — or the role does not ship
```

Каждая роль следует одному и тому же контракту, закреплённому спецификацией и CI:

1. **Три теста на готовность к выпуску** — практик, читая роль, должен кивать, а не пожимать плечами; агент с этой ролью должен принимать измеримо иные решения, чем без неё; ничто в роли не должно выводиться из одного названия должности.
2. **Фиксированная анатомия** — идентичность, первопринципное ядро, условные эвристики («при X по умолчанию делай Y, если не Z»), исполняемый фреймворк принятия решений, типичные ошибки и разобранный пример с реальными, сходящимися числами, заканчивающийся настоящим итоговым результатом (меморандумом, правкой договора, отчётом).
3. **Тройка справочных материалов** — файл с подробным playbook'ом/артефактами и заполненными шаблонами, `red-flags.md` (сигнал → что он означает → первый вопрос → какие данные запросить) и `vocabulary.md` (профессиональные термины с указанием типичного неверного употребления).
4. **Происхождение (Provenance)** — источники названы поимённо; конкретные числа либо прослеживаются до источника, либо помечены как заявленная эвристика. Регулируемые роли (право, медицина, финансы) несут явные оговорки.
5. **Костяк O*NET** — покрытие следует классификации профессий Министерства труда США (1016 профессий), поэтому рост систематический, а не по принципу «что показалось интересным на этой неделе».

Полная спецификация, рубрика и конвейер составления черновиков LLM: [`AUTHORING.md`](./AUTHORING.md).

## Как мы проверяем — прозрачно, без необходимости верить на слово

«Написано экспертами» — это утверждение; этот репозиторий вместо этого предоставляет доказательства. Четыре независимых уровня, каждый из которых может запустить кто угодно из этого чекаута:

```
 layer 1  SOURCING      every threshold traces to a named practitioner
 (input)                source (books, standards, postmortems) or is
                        labeled a stated heuristic — see each role's
                        Sources section
 layer 2  MECHANICAL    scripts/lint_roles.py on every PR: schema,
 (CI)                   required sections, references/ trio, banned
                        filler phrases, real numbers in the worked
                        example — generic text fails the build
 layer 3  COUNTERFACTUAL evals/: same model answers the same scenario
 (measured)             with and without the role, blind judge scores
                        observable expert behaviors in random A/B order
 layer 4  PARITY        evals/parity/: skill answers real questions that
 (measured)             real practitioners already answered publicly —
                        blind judge compares head-to-head
```

Последние опубликованные прогоны (2026-07-06, отвечает Haiku 4.5, судит вслепую Sonnet 5, обе тестовые обвязки):

- **Контрфактическое сравнение:** роль побеждает в **13 из 15 сценариев** (1 ничья, 1 поражение) — 72% критериев экспертного поведения выполнено против 37% у базового варианта без роли.
- **Паритет с людьми:** ответ роли предпочли реальному, принятому ответу практика на Stack Exchange в **5 из 8** слепых прямых сравнений (выборка небольшая; наборы вопросов растут — воспринимайте это как сигнал, а не как исследование).

Каждый результат воспроизводим: `python3 evals/run_evals.py` и `python3 evals/parity/run_parity.py`. Если роль не проходит эти тесты, это тоже становится публичным — суть в измерении, а не в маркетинге. Проверка практиком остаётся золотой звездой сверху (`metadata.maturity`), но нижняя планка доверия измеряется, а не гарантируется на слово.

## Использование с вашим AI-инструментом

### Без настройки: вставьте это в своего агента

Скопируйте это в Claude Code, Codex, Cursor или любого агента с доступом к shell, опишите свою задачу в конце — и агент сам установит нужного эксперта:

```text
Install a domain expert for my task from the open-source library
https://github.com/wonsukchoi/domain-experts :

1. Run: npx --yes domain-experts match "<my task>" --json
2. If it returns a confident match, install it:
   npx --yes domain-experts add <slug>
   (default target is ./.claude/skills/<slug>; if you are not Claude Code,
   pass --to <your skills directory>/<slug>, e.g. --to .codex/skills/<slug>)
3. Read the installed SKILL.md fully. Open files under references/ whenever
   the task needs the depth they cover. Then do my task reasoning as that
   expert — apply its thresholds, red flags, and decision framework.
4. If there is no confident match, tell me which roles came closest and
   continue as a generalist — do not pretend to be an expert the library
   does not have.

My task: <describe your task here>
```

**Пользователи Claude Code, Codex, Gemini CLI, Cursor, Windsurf, Roo Code и Amp:** пропустите вставку текста — `npx domain-experts command --tool <id>` один раз устанавливает `/domain-expert`, а дальше просто запускайте `/domain-expert <task>` каждый раз (см. раздел [слэш-команда `/domain-expert`](#domain-expert-slash-command) ниже).

### Установка под конкретный инструмент

| Инструмент | Как |
|---|---|
| **Claude Code** | `npx domain-experts add <slug>` — попадает в `./.claude/skills/<slug>/`, автоматически распознаётся как скилл. |
| **Codex CLI** | Та же команда с `--to .codex/skills/<slug>` (проект) или `--to ~/.codex/skills/<slug>` (личный уровень). Новая сессия подхватит её автоматически. |
| **Cursor** | Та же команда с `--to .cursor/skills/<slug>` — Cursor читает тот же формат `SKILL.md` нативно. |
| **Windsurf, Roo Code, Goose и другие совместимые с SKILL.md инструменты** | Та же команда с `--to <каталог скиллов инструмента>/<slug>` — точный путь смотрите в документации своего инструмента. |
| **Инструменты, читающие `AGENTS.md`** (GitHub Copilot, Jules, Amp, Zed и др.) | Установите куда угодно в репозитории (например, `--to skills/<slug>`), затем добавьте одну строку в `AGENTS.md`: `When a task needs <role> judgment, read skills/<slug>/SKILL.md first.` |
| **Любой чат-AI без shell** | Откройте роль на GitHub, вставьте `SKILL.md` в системный промпт или кастомные инструкции; вставляйте файлы из `references/`, когда диалогу нужна эта глубина. |

Каждая установка копирует роль целиком — `SKILL.md` плюс `references/` — так что подробные playbook'и путешествуют вместе с ней.

### Автоматическая диспетчеризация

[`skills/domain-expert-router/SKILL.md`](./skills/domain-expert-router/SKILL.md) — это мета-скилл, убирающий даже шаг `match`: установите его командой `npx domain-experts add domain-expert-router`, загрузите один раз, и ваш агент сам найдёт нужную роль для запросов вида «веди себя как X» и честно скажет, если подходящей роли ещё нет.

### Слэш-команда `/domain-expert`

```sh
npx domain-experts command --tool <id>   # claude (default), codex, gemini, cursor, windsurf, roo, amp
```

Перезапустите сессию, затем используйте `/domain-expert <task>` напрямую — например, `/domain-expert review this vendor contract`. Команда запускает `match`, загружает `SKILL.md` победившей роли (и `references/` при необходимости) и отвечает как этот эксперт — либо честно говорит, что подходящего пока нет. Та же идея, что и у скилла-роутера выше, только упакованная как одноразовая команда, а не постоянно загруженный скилл.

| `--tool` | Устанавливается в | Примечания |
|---|---|---|
| `claude` (по умолчанию) | `.claude/commands/domain-expert.md` | |
| `codex` | `~/.codex/prompts/domain-expert.md` | Codex читает промпты только из пользовательского каталога, локального для проекта варианта нет; в документации OpenAI этот механизм помечен устаревшим в пользу «skills», но пока работает |
| `gemini` | `.gemini/commands/domain-expert.toml` | Формат TOML |
| `cursor` | `.cursor/commands/domain-expert.md` | |
| `windsurf` | `.windsurf/workflows/domain-expert.md` | В Windsurf это называется «workflows» |
| `roo` | `.roo/commands/domain-expert.md` | |
| `amp` | `.agents/commands/domain-expert.md` | Расположение в Amp фиксировано в корне репозитория, отдельного глобального каталога нет |

Добавьте `--global`, чтобы установить в пользовательский каталог инструмента (например, `~/.claude/commands/`, `~/.cursor/commands/`) вместо каталога проекта, либо `--to <path>` для полностью произвольного расположения.

### Справочник по CLI

```sh
npx domain-experts list          # browse all roles
npx domain-experts search lawyer # substring search
npx domain-experts match "review this like our CFO" [--json]
npx domain-experts add <slug> [--to dir]
npx domain-experts command [--tool <id>] [--global] [--to path]  # install the /domain-expert command
```

`match` оценивает роли по пересечению ключевых слов и сообщает либо уверенное совпадение, либо кандидатов с низкой уверенностью, либо честное «пока не покрыто» — вместо того чтобы молча угадывать. `--json` — для программного использования.

npm-пакет фиксирует снимок библиотеки ролей на момент каждого релиза. Для неопубликованного самого свежего состояния используйте `npx --yes github:wonsukchoi/domain-experts <command>` — тот же CLI, прямо из `main`.

## Как внести вклад — этот репозиторий растёт по принципу сложного процента

Каждая добавленная роль делает роутер умнее, каждая правка доходит до всех пользователей при следующем релизе, а каждый вопрос из evals делает планку качества труднее подделать. Промпт, который вы написали для себя, умирает вместе с вашей сессией; роль, которую вы добавили сюда, работает для всех, вечно, и продолжает улучшаться после того, как вы уйдёте. В этом вся ставка: **1016 профессий — это не сольный проект, а общий фонд (commons).**

Частые вопросы (ошибки линтера, конфликты при push, процесс релиза) → [`docs/FAQ.md`](./docs/FAQ.md).

Три способа поучаствовать, при любом уровне навыков:

1. **Вы работаете в роли, которую мы уже покрываем?** Прочитайте её. Любая неточность — это 2-минутный [issue с правкой от практика](../../issues/new/choose) — самый ценный вклад, который может получить этот проект. Навыки работы с PR не нужны.
2. **Хотите написать или улучшить роль?** Следуйте точному рецепту в [`CONTRIBUTING.md`](./CONTRIBUTING.md) — он написан настолько точно, что его может выполнить LLM, так что вы и ваш AI-ассистент можете сделать это вместе. Линтер сообщит, если структура не дотягивает до планки, ещё до того, как её увидит человек. 42 устаревшие роли [можно забрать прямо сейчас](../../issues/1).
3. **Не умеете писать, но умеете находить?** Собирайте вопросы для сравнения паритета (`evals/parity/harvest_stackexchange.py`) или подайте [запрос на роль](../../issues/new/choose) с задачами, которые вы бы ей делегировали.

Если спецификация противоречит реальности практика, спецификация проигрывает — напишите об этом в своём PR, и мы её исправим.
